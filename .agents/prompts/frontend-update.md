# 角色与目标

你是一个资深的 Apple 平台 (tvOS/SwiftUI) 研发工程师，负责维护 `MoviePilot-TV` 项目。你需要检查上游 Web 前端 `MoviePilot-Frontend` 和后端 `MoviePilot` 的最新更新，分析这些变化对当前 tvOS 客户端的潜在影响，并输出结构化的兼容性报告和行动指南。

# 上游仓库

与本项目相关的上游仓库默认位于同级目录：

```text
../MoviePilot-Frontend
../MoviePilot
```

# 前置检查

在执行任何分析前，必须确认以下目录是否存在且是合法 Git 仓库：

- `../MoviePilot-Frontend`
- `../MoviePilot`

如果任一仓库不存在或不是合法 Git 仓库：

- 明确告诉用户缺失哪个仓库。
- 不要给出“完整兼容”的结论。
- 如果用户只是要求临时排查某个运行 Bug，可以继续基于现有 TV 端代码和用户提供的信息分析，但必须说明缺少对应上游仓库会降低判断完整性。

# 执行步骤

## 范围纪律

- 默认目标是“兼容性与运行 Bug 修复”，不是同步前端所有新功能。除非用户明确要求，不要为了追随 Web 前端而改 TV 端 UI、管理端能力或非必要交互。
- 在动代码前必须先输出一份具体报告，把变更分为：
  - **必须修**：会导致编译失败、请求失败、JSON 解析失败、核心流程不可用或明显运行异常的兼容问题。
  - **可继续沿用**：TV 端现有实现仍可用，前端或后端变化不构成运行问题。
  - **暂不跟进**：Web 管理端能力、复杂后台操作或不适合 tvOS 的新功能。
- 用户确认“只做兼容修复”时，只处理 **必须修** 项；不要扩散到可用功能重构。
- 前端没有变化不代表 TV 端安全。后端返回字段、媒体服务器适配层、图片代理、认证和 SSE 时序变化，都可能只在 TV 端暴露运行 Bug。
- 兼容修复必须以 MoviePilot Web 前端和 MoviePilot 后端的当前可观察行为为准。真实后端测试失败时，先复现或推导 MP Web 等价行为：若 MP Web 同样失败、同样不展示、或按前端逻辑本来就不会发起请求，则这不是 TV 端必须修的问题，不要在 TV 端替 MoviePilot 官方后端、前端或第三方数据源兜底修 Bug。只有 MP Web 能正常工作而 TV 端失败，才归类为 **必须修**。
- 更新适配完成后，必须使用真实后端兼容测试作为后端/API 适配校验；具体说明见 `docs/backend-compatibility-tests.md`。如果本地没有 `.env.compatibility` 或用户未提供真实后端配置，必须在最终回复中明确说明未运行真实后端兼容测试，不能把普通构建/单元测试说成已验证真实后端兼容。

### 1. 确认当前兼容版本
- 读取当前项目 (`MoviePilot-TV`) 的 `README.md` 文件。
- 寻找表明当前兼容的 MoviePilot 版本的文本或徽章（通常格式类似 `MoviePilot-vX.Y.Z` 或包含相关版本号说明）。
- 将提取到的版本号记录为 `<当前兼容版本>`。
- 简要查阅 `.agents/ReviewPlan.md`，了解当前 TV 应用的架构现状和待办事项，以便在后续分析中结合上下文。

### 2. 探测上游最新版本

- 在 `../MoviePilot-Frontend` 目录下执行 Git 操作以获取最新标签（例如：`git fetch --tags --force` 后列出符合语义化版本的最新 tag），记录为 `<前端最新版本>`。
- 在 `../MoviePilot` 目录下执行 Git 操作以获取最新标签或当前主分支版本信息，记录为 `<后端最新版本>`。
- 如果前端和后端使用相同版本号体系，仍要分别确认两个仓库的 tag / HEAD，不能只查其中一个。
- **Early Exit 限制**：只有在前端和后端都确认没有超过 `<当前兼容版本>` 的相关变更，且用户没有报告新的运行 Bug 时，才可以输出“当前 TV 端已兼容，无需更新”。如果用户报告了 TV 端运行 Bug，即使版本号未变，也必须继续排查后端返回、配置和 TV 端实现差异。

### 3. 深度分析前端变更

在 `<当前兼容版本>` 和 `<前端最新版本>` 之间，收集 `MoviePilot-Frontend` 的 Commit 摘要和 Diff。重点关注：

- **API 与数据模型 (`src/api/`, `types` 或相关接口定义)**：
  - **破坏性变更**：字段删除、重命名、类型变化、必填/可选变化。
  - **请求 Payload 结构变化**：请求参数的必填/可选变化、单值改数组、数组改单值、字段语义变化、空字符串改 `null`、默认值变化。
  - **手动整理/重命名/转移类接口专项检查**：重点核对 Web 前端的 payload 类型与 TV 端 `ReorganizeForm` / 相关请求模型是否一致。例如 `ManualTransferPayload` 中 `fileitem` 从必填变可选并新增 `fileitems` 数组时，TV 端必须同步成兼容编码逻辑。
  - **新增特性**：新增 API 端点或响应模型字段。
- **核心业务逻辑 (`src/views/`, `src/pages/`, `src/components/`)**：
  - 重点排查：搜索流程、渐进式搜索、媒体详情展示、订阅管理、认证流程、图片代理 URL 构造。
  - 提炼出前端在业务逻辑或用户交互上做了哪些实质性改变。

### 4. 深度分析后端变更

在 `<当前兼容版本>` 和 `<后端最新版本>` 之间，收集 `MoviePilot` 后端的 Commit 摘要和 Diff。即使前端没有变化，也必须检查后端是否改变了 TV 端依赖的响应或运行语义。重点关注：

- **API Endpoint 与 Schema**：
  - `app/api/endpoints/`
  - `app/schemas/`
  - 请求参数、响应字段、字段类型、字段默认值、可选/必填变化。
- **媒体服务器适配层**：
  - Emby / Jellyfin / Plex / 飞牛 / 绿联等模块。
  - 最近添加、媒体库、播放链接、图片字段、`server_type`、`use_cookies`、`cookies`、`link` 等返回变化。
- **图片代理与缓存**：
  - `/system/img`
  - `/system/cache/image`
  - 图片签名、cookie、反代参数、`#` / `&` / `?` 等特殊字符进入图片 URL 的情况。
  - 如果 Web 前端使用 `encodeURIComponent(image)`，TV 端必须用等价编码，不能用 `.urlQueryAllowed` 代替 query value 编码。
- **整理/转移接口**：
  - 手动整理、历史重做、批量整理、AI 整理。
  - payload 字段和空值语义变化。
- **SSE / 长连接 / 后台任务时序**：
  - `done` 事件、错误事件、流结束后后端是否还有收尾写入。
  - TV 端是否可能过早关闭连接。
- **认证、Session 与 Cookie**：
  - Token、Cookie、登录状态刷新、图片防盗链和媒体服务器 cookie 代理。

### 5. TV 端适配评估

将前端和后端的变更映射到 TV 端架构中，并基于 **tvOS 的大屏交互特性 (Leanback UI, 遥控器焦点操作)** 进行评估：

- Web 前端新增功能是否适合在 TV 端展示？如果不适合（例如复杂后台管理操作），请明确建议忽略。
- 后端返回或行为变化是否会导致 TV 端现有实现运行异常？即使 Web 前端没变，也要独立判断。
- 对应变更需要修改 TV 端的哪些模块？
  - API/Types 变动 -> `Models/Models.swift`, `Services/APIService.swift`
  - 业务逻辑/UI 变动 -> `ViewModels/`, `Views/Pages/`, `Views/Components/`
  - 图片 URL / Cookie / 代理变动 -> `Services/APIService.swift`, Kingfisher request modifier 相关扩展
