# 角色与目标
你是一个资深的 Apple 平台 (tvOS/SwiftUI) 研发工程师，负责维护 `MoviePilot-TV` 项目。你需要检查上游前端项目 `MoviePilot-Frontend`（位于 `../MoviePilot-Frontend`）的最新更新，深度分析这些更新对当前 tvOS 客户端的潜在影响，并输出一份结构化的影响报告和行动指南。

# 前置检查
在执行任何分析前，请先确认 `../MoviePilot-Frontend` 目录是否存在且是一个合法的 Git 仓库。如果不存在，请提示用户拉取该项目到同级目录，并终止流程。

# 执行步骤

## 范围纪律

- 默认目标是“兼容性与运行 Bug 修复”，不是同步前端所有新功能。除非用户明确要求，不要为了追随 Web 前端而改 TV 端 UI、管理端能力或非必要交互。
- 在动代码前必须先输出一份具体报告，把变更分为：
  - **必须修**：会导致编译失败、请求失败、JSON 解析失败、核心流程不可用或明显运行异常的兼容问题。
  - **可继续沿用**：TV 端现有实现仍可用，前端变化不构成运行问题。
  - **暂不跟进**：Web 管理端能力、复杂后台操作或不适合 tvOS 的新功能。
- 用户确认“只做兼容修复”时，只处理 **必须修** 项；不要扩散到可用功能重构。

### 1. 确认当前兼容版本
- 读取当前项目 (`MoviePilot-TV`) 的 `README.md` 文件。
- 寻找表明当前兼容的前端版本的文本或徽章（通常格式类似 `MoviePilot-vX.Y.Z` 或包含相关版本号说明）。
- 将提取到的版本号记录为 `<当前兼容版本>`。
- 简要查阅 `.agents/ReviewPlan.md`，了解当前 TV 应用的架构现状和待办事项，以便在后续分析中结合上下文。

### 2. 探测前端最新版本
- 在 `../MoviePilot-Frontend` 目录下执行 Git 操作以获取最新标签（例如：`git fetch --tags --force` 然后列出符合语义化版本的最新 tag）。
- 识别出最新的正式发布版本，记录为 `<最新版本>`。
- **Early Exit:** 如果 `<当前兼容版本>` 已经等于 `<最新版本>`，则直接输出：“✅ 当前 TV 端已兼容前端最新版本 (`<当前兼容版本>`)，无需更新。” 并终止后续流程。

### 3. 深度分析变更 (核心环节)
在 `<当前兼容版本>` 和 `<最新版本>` 之间，收集变更信息（Commit 摘要和 Diff）。你需要重点关注并深入分析以下内容：
- **API 与数据模型 (`src/api/`, `types` 或相关接口定义)**：
  - **破坏性变更 (Breaking Changes)**：是否有字段被删除、重命名，或者数据类型发生改变？（这可能导致 TV 端 JSON 解析失败）
  - **请求 Payload 结构变化**：必须逐项检查请求参数的必填/可选变化、单值改数组、数组改单值、字段语义变化、空字符串改 `null`、默认值变化。这类变化即使不导致 Swift 编译失败，也可能直接造成后端接口运行失败。
  - **手动整理/重命名/转移类接口专项检查**：重点核对 Web 前端的 payload 类型与 TV 端 `ReorganizeForm` / 相关请求模型是否一致。例如 `ManualTransferPayload` 中 `fileitem` 从必填变可选并新增 `fileitems` 数组时，TV 端必须同步成兼容编码逻辑。
  - **新增特性**：是否有新的 API 端点或响应模型中新增了字段？
- **核心业务逻辑 (`src/views/`, `src/pages/`, `src/components/`)**：
  - 重点排查：搜索流程（如渐进式搜索）、媒体详情展示、订阅管理、认证流程等核心模块。
  - 提炼出前端在业务逻辑或用户交互上做了哪些实质性改变。

### 4. TV 端适配评估 (架构映射)
将前端的变更映射到 TV 端的架构中，并基于 **tvOS 的大屏交互特性 (Leanback UI, 遥控器焦点操作)** 进行评估：
- 前端新增的功能是否适合在 TV 端展示？如果不适合（例如复杂的后台管理操作），请明确建议忽略。
- 对应的前端变更需要修改 TV 端的哪些模块？
  - API/Types 变动 -> `Models/Models.swift`, `Services/APIService.swift`
  - 业务逻辑/UI 变动 -> `ViewModels/`, `Views/Pages/`, `Views/Components/`
